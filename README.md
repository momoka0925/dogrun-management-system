
# DogRun Manager

ドッグラン＆BBQ施設向けに開発した、顧客管理・犬管理・予約管理アプリケーションです。
「現場のスタッフが毎日使う業務台帳」をコンセプトに、UI設計からデータベース設計、CRUD実装までを一通り構築しました。

デモ環境:[https://dogrun-management-system.vercel.app/]
(テストデータで動作しています)


## 概要

小規模事業者がスプレッドシートや紙の台帳で管理していた顧客情報・犬の情報・予約状況を、Web上で一元管理できるようにしたアプリです。

構成としては以下の3画面に分かれています。

トップページ（`/`）では顧客の一覧・登録・編集・削除・検索ができます。
`/dogs` では犬の情報を管理し、顧客とのリレーションを持たせています。
`/reservations` では予約の登録・編集・削除に加えて、ステータス（仮予約・確定・キャンセル）の管理ができます。


## 技術スタック

フロントエンドはNext.js（App Router）とTypeScriptで構築しています。
App Routerを採用した理由は、ファイルベースのルーティングによって画面構成がそのままディレクトリ構造に反映され、機能追加のたびにディレクトリを増やしていくだけで自然と整理された構成を保てる点です。

スタイリングはTailwind CSSを使用しています。コンポーネントごとにクラスを当てていく方式は、最初は冗長に見えましたが、結果的に「このコンポーネントの見た目はこのファイルだけ見ればわかる」状態を作れたのでメンテナンス性は高いと感じています。

バックエンドはSupabase（PostgreSQL）です。テーブル設計・REST API・認証・RLS（行レベルセキュリティ）が一つのサービスに統合されているため、個人開発でバックエンドをゼロから構築する手間を大きく省けました。

デプロイはVercelを使用しています。


## 実装した機能

### 顧客管理

顧客の新規登録、一覧表示、編集、削除ができます。
名前・電話番号・メールアドレスでのリアルタイム検索に対応しており、入力するたびに一覧がフィルタリングされます。
有効・無効のステータスも持たせており、退会済みの顧客を一覧から除外するような使い方も想定しています。

編集中の行はテーブル上でハイライト表示されるようにしており、「今どの行を編集しているか」が一目でわかるようにしています。

### 犬管理

犬の情報を顧客に紐付けて管理する機能です。
`dogs` テーブルに `customer_id` という外部キーを持たせ、Supabaseの `select("*, customers(name)")` というJOINクエリを使ってオーナー名を一緒に取得しています。

犬のサイズ（小型・大型）やワクチン接種状況をバッジで表示し、登録数・小型犬の数・ワクチン接種済みの数などをサマリーカードでリアルタイム集計しています。

### 予約管理

このアプリの中核となる機能です。
`reservations` テーブルを新設し、顧客・犬と紐付けています。

ステータスは「仮予約」「確定」「キャンセル」の3種類で、これをアプリ側のif文だけで制御するのではなく、SupabaseのCHECK制約を使ってデータベース側でも値を保証するようにしました。
不正な値が入る可能性をDBレベルで排除しています。

一覧画面ではステータスごとのフィルタリングとキーワード検索を組み合わせられるようにし、日付の降順でソートしています。
PCでは表形式、スマホではカード形式に切り替わるよう、レイアウトを完全に分けて実装しました。

### UI全体について

見た目については、よくあるSaaSダッシュボードのテンプレート的な雰囲気を避け、深緑とベージュを基調にした「業務台帳」のようなデザインにしています。
情報密度を上げつつも、ローディング中・データが0件のとき・エラーが発生したときの表示をすべての画面で個別に作り込んでいます。

フォームはアコーディオン形式にしており、一覧を見ながら新規登録・編集ができるようにしています。


## 開発の進め方について

このプロジェクトはClaudeをペアプログラマーとして使いながら開発しました。
ただ、コードを生成させて貼るだけではなく、一つ一つの実装について「なぜそう書くのか」を確認しながら進めることを意識していました。

例えば `useEffect(() => {...}, [])` の第二引数を空配列にする理由（無限ループを防ぐため）や、SupabaseのJOINがSQLのLEFT JOINとどう対応しているか、RLSがアプリ側の条件分岐とは違いデータベース側で強制される仕組みであることなど、コピーする前に必ず説明を求め、自分の言葉で説明できる状態にしてから次に進むようにしました。

開発中にいくつかの問題にも当たりました。

`dogs` テーブルを参照するページを実装した際、Supabase側にテーブルそのものを作成していなかったために `Could not find the table 'public.dogs'` というエラーが出ました。これは単純なミスでしたが、テーブル定義のSQLを見直すきっかけになり、`on delete cascade` を含めたリレーション設計を改めて確認しました。

また、ボタンの視認性が低いという指摘を受けて修正した際は、単に色を変えるだけでなく、背景色とテキスト色のコントラストを意識した配色に直す、という流れで対応しました。

機能を顧客管理から犬管理、予約管理へと拡張していく過程では、既存のコンポーネント構成（フォームとテーブルをそれぞれ別ファイルに分離するパターン）を新しい機能でも踏襲するようにしています。型定義についても、`types/customer.ts` で採用した「DBから返ってくる型」と「フォーム入力用の型」を分けるという考え方を、`types/reservation.ts` でも引き継ぎました。

最終的には、現状の構成を踏まえたうえで、お客様用のアプリと店舗管理用のアプリを完全に分離する構成や、Supabase Authを使ったログイン状態に応じた入力内容の自動反映、`anon key` と `service_role key` の権限分離についても検討しました。

## データベース設計

顧客（customers）を中心に、犬（dogs）と予約（reservations）がそれぞれ外部キーで紐付く構成です。

```
customers
   |
   |---- dogs            （customer_id で紐付け）
   |
   `---- reservations    （customer_id で紐付け、status は pending / confirmed / cancelled）
```

顧客を削除した場合、紐付く犬・予約も `on delete cascade` によって一緒に削除されます。
すべてのテーブルにRLSを設定済みで、今後の権限分離（スタッフ用・顧客用のアクセス制御）を見据えた構成にしています。
